# StreamTrans 同传 App 规划（移动端 · 语音→文本字幕 · 端侧推理）

> 形态：外部 ASR 转文字 → 本文模型流式翻译 → 屏幕实时显示译文字幕。
> 平台：移动端（Android 先行，端侧推理）。语音输入由外部 ASR 提供（项目定位）。
> 本规划是 Phase 3（wait-k 流式）+ Phase 4（端侧导出）的**产品化落地**——软件依赖这两阶段产物，
> 故分"模型侧前置"与"App 侧工程"两条线，并行推进、在端侧推理接口处汇合。

---

## 0. 端到端数据流

```
麦克风音频
  → [ASR 引擎] 流式 partial/final 转写(中或英)
  → [分句/触发器] 决定何时把新增前缀喂给翻译
  → [StreamTrans 流式翻译(wait-k)] 增量产出译文 token
  → [字幕渲染] 增量上屏 + ASR 修订时的回改处理
```

关键难点不在单点模型，而在**流式拼接**：ASR partial 会回改、翻译要增量且稳定、延迟要可控。

## 1. 模型侧前置（依赖 Phase 3/4，必须先于 App 完成）

### 1.1 Phase 3：wait-k 流式翻译
- prefix-to-prefix 训练：读到源前缀 k 个单位即开始输出，固定 k 跑通全链路。
- on-policy OPD 叠加（reverse KL + 师生 top-k 并集），缓解流式逐前缀解码的暴露偏差。
- 产出**流式解码器**：支持"喂入增量源前缀 → 吐出增量目标"的接口（而非整句 generate）。
- 调出可接受的延迟-质量工作点（AL/LAAL vs BLEU）。

### 1.2 Phase 4：端侧导出
- 量化（优先 int4/int8）；产出 MNN 或 llama.cpp（GGUF）格式。
- **混合架构落地验证**：线性注意力（Gated DeltaNet/Mamba 类）算子在目标框架/版本上的正确性与性能（已核实 llama.cpp `ggml_gated_delta_net` / MNN 3.4.1 支持，需端侧实测）。
- 产出端侧推理库的**流式增量接口**（带 KV/状态保持，逐前缀喂入）。

## 2. App 侧工程（可与模型侧并行）

### 2.1 技术栈选型（首版 Android）
- 推理：MNN（移动端成熟、线性注意力有算子）或 llama.cpp（GGUF + JNI）——按 Phase 4 实测延迟/体积定。
- **ASR：whisper.cpp（tiny/base 量化版，端侧离线）。已定，非系统 ASR。**
  - 理由（按重要性）：① 行为固定、参数可控、论文延迟数据可干净归因（系统 ASR 回改/断句是黑盒且机型不一）；
    ② 连续长时同传无 OS 会话限制（系统 ASR 静音自动断句/超时，长会议要不停对抗会话重启）；
    ③ 跨平台一套 C++ 实现（系统 ASR 要安卓+iOS 两套语义不同的适配器）；④ 一定端侧离线，"纯端侧"论点干净。
  - 代价与化解：资源叠加（ASR+翻译抢内存/算力）→ 用 tiny/base 量化（~75–150MB），算进端侧**总**预算，
    反而让论文 §4.5"端侧总开销"更完整；需自写音频采集管线 → 一次性工程、可复用。
  - 退路：仅当目标 app 是一次性答辩 demo、不进论文实测、不要连续长时，才考虑系统 ASR。
  - 抽象层 `StreamingTranscriber`（feed_audio/poll_partial_final/reset）仍保留，便于替换，但首版只接 whisper.cpp。
- UI：原生（Kotlin）或跨端（Flutter）——首版求快可原生。

### 2.2 流式拼接核心（最难、最该先验证）
- **ASR 修订处理**：partial 转写会改写尾部；策略——只把"已稳定的前缀"喂翻译（如 final 段或 N 字未变的前缀），尾部不稳定段缓冲。
- **增量翻译触发**：按"新增稳定源单位达 k"触发一次流式解码步（与 wait-k 的 k 对齐）。
- **字幕回改**：源回改导致已出译文需修正时，UI 支持尾部译文的"软回改"（划线/淡出替换），或采用"只追加不回改"的保守策略（延迟换稳定）。
- **延迟预算**：端到端延迟 = ASR 延迟 + 翻译首字延迟 + 渲染；逐项测、定预算。

### 2.3 工程脚手架
- 推理接口抽象层：`StreamingTranslator`（init/feed_source_prefix/poll_target/reset），屏蔽 MNN vs llama.cpp。
- 词表桥接：端侧也需 `VocabRemapper` 等价逻辑（new-id ↔ 文本），与训练侧一致。
- 配置：k、稳定性阈值、最大缓冲、回改策略，全部可调。

## 3. 里程碑（建议顺序）

| 阶段 | 目标 | 产出 | 依赖 |
|---|---|---|---|
| M0 | 离线整句端侧推理跑通 | 量化模型在手机上整句翻译出结果(非流式) | Phase 4 导出 |
| M1 | 流式解码器(服务端先) | 喂增量前缀→吐增量译文,延迟-质量曲线 | Phase 3 |
| M2 | 端侧流式推理 | 流式解码器移植到端侧推理库,带状态保持 | M0+M1 |
| M3 | ASR 接入 + 拼接 | 流式转写→稳定前缀→增量翻译,字幕上屏 | M2 |
| M4 | App demo | Android app:听中文/英文,屏幕实时双语字幕,可调 k | M3 |

### 3.1 M3 子任务拆解（基于 whisper.cpp）

M3 是全项目工程风险最高的一环——难点全在"流式拼接"。按依赖顺序拆 8 个子任务：

| # | 子任务 | 内容 | 关键风险 / 产出 |
|---|---|---|---|
| T1 | 音频采集管线 | `AudioRecord` 16kHz 单声道 PCM → 环形缓冲，连续不断流；后台线程 + 背压处理 | 一次性工程，可复用；产出连续 PCM 流 |
| T2 | whisper.cpp 集成 | JNI 封装，模型加载（tiny/base 量化），暴露"喂 PCM 段 → 取转写"接口 | 模型体积/加载耗时；产出 `StreamingTranscriber` 实现 |
| T3 | 分窗/VAD 策略 | 滑动窗 + 重叠喂 whisper（whisper 非天然流式，靠重叠窗近似）；可选 VAD 切静音 | 窗长/重叠 = 延迟 vs 稳定的权衡；**直接决定回改频率** |
| T4 | 稳定前缀提取 | 从"会回改的转写流"判定哪段已稳定可喂翻译（如重叠窗中 N 次未变的前缀视为稳定） | M3 算法核心；稳定判太松→错译，太紧→延迟 |
| T5 | 增量翻译触发 | 稳定源单位每新增 k 个 → 调 M1 的流式解码器走一步，取增量译文 | 与 wait-k 的 k 对齐；触发粒度影响延迟 |
| T6 | 字幕渲染 + 回改 | 译文增量上屏；源回改导致译文需改时：尾部"软回改"(替换/淡出) 或"只追加不回改"(延迟换稳定) | 体验成败点；两种策略都实现、可切 |
| T7 | 端到端延迟埋点 | 分段打点：ASR 延迟 / 翻译首字延迟 / 渲染延迟 / 总延迟（AL、LAAL） | **直接产出论文 §4.4/§4.5 数据** |
| T8 | 配置与可调 | k、稳定阈值、窗长/重叠、缓冲上限、回改策略全部可调，便于调体验和做实验 | 支撑延迟-质量权衡实验 |

依赖与顺序：T1→T2 是基础设施（音频+ASR 接入）；**T3+T4 是拼接核心、风险最高，建议最早在 M1 阶段就用真实 whisper 输出预演**（不等 M3，避免后期才暴露工作量）；T5 接 M1 的流式解码器；T6 是体验关键、需 A/B 两种回改策略；T7 贯穿全程，是论文硬数据来源；T8 收尾，把所有参数暴露出来做实验。

注：whisper **不是天然流式**——它是整段编码器模型，端侧"流式"靠 T3 的重叠滑动窗近似，这是 whisper.cpp 路线的固有特性，T3/T4 的设计要围绕这一点（与真流式 ASR 的回改模式不同）。

## 4. 风险与未决

- **流式拼接体验** > 模型质量：ASR 回改与译文稳定性的权衡，是体验成败关键，建议 M1 就用真实 ASR 输出做拼接验证，别等到 M3。
- **端侧线性注意力性能**：算子"支持"≠"够快"，Phase 4 需实测；不达标的退路是 full-attention 段更多或服务端兜底。
- **延迟下限**：wait-k 的 k 越小延迟越低但质量越差；端侧推理本身的单步延迟也进预算，可能逼大 k。
- **ASR 选型**：端侧中英 ASR 的质量/延迟/体积，会直接决定整体体验，需早定。

## 5. 与论文的关系
M1（延迟-质量曲线）与 M2/M4（端侧延迟/内存实测）正是论文 §4.4/§4.5 的实验数据来源——
**软件落地与论文实验是同一批工作**，优先做能同时产出论文数据的里程碑（M1、M2）。
